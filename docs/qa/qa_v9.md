# PDF 청킹 품질 검증 테스트 계획

## 1. 배경: 왜 PDF를 "청킹"하는가?

### 1.1 문제 상황

my-atlas의 Senior 채팅(`/senior`)은 AI에게 질문하면 답변을 해주는 기능이다.
하지만 AI는 우리가 올린 ISTQB 교재의 내용을 **원래 알지 못한다.**
AI가 교재 내용을 참고해서 답변하려면, 우리가 교재 내용을 AI에게 **전달**해야 한다.

그런데 문제가 있다. ISTQB CTFL 교재는 72페이지짜리 문서인데,
AI에게 한 번에 보낼 수 있는 텍스트 양에는 한계가 있다.
**교재 전체를 매번 통째로 보내는 것은 비효율적**이다.

### 1.2 해결 방법: 청킹 + 검색

그래서 이런 방식을 사용한다:

```
[준비 단계 — PDF 업로드 시 한 번만 실행]

  72페이지 PDF
       ↓
  텍스트 추출 (PDF에서 글자만 뽑아냄)
       ↓
  "챕터/절" 단위로 구간 나누기 (섹션 파싱)
       ↓
  각 구간을 적당한 크기의 조각으로 자르기 (청킹)
       ↓
  각 조각의 "의미"를 숫자 배열로 변환 (임베딩 — 아래 1.3에서 설명)
       ↓
  데이터베이스에 저장
```

```
[사용 단계 — 사용자가 질문할 때마다 실행]

  사용자 질문: "테스팅의 7원칙이 뭐야?"
       ↓
  질문의 "의미"를 숫자 배열로 변환
       ↓
  DB에 저장된 조각들 중 질문과 "의미가 비슷한" 조각 2개를 찾음
       ↓
  찾은 조각을 AI에게 "참고 자료"로 전달
       ↓
  AI가 참고 자료를 읽고 답변 생성
```

이 전체 과정을 **RAG(Retrieval Augmented Generation)** 이라고 부른다.
직역하면 "검색으로 보강된 생성" — AI가 답변을 **생성**하기 전에, 관련 자료를 **검색**해서 **보강**한다는 뜻이다.

### 1.3 "임베딩"이란?

임베딩은 **텍스트의 의미를 숫자 배열로 바꾸는 것**이다.

비유하자면 도서관의 분류 번호와 비슷하다:
- "경계값 분석 기법" → `[0.82, -0.15, 0.43, ...]` (1,536개의 숫자)
- "동등 분할 기법" → `[0.79, -0.12, 0.41, ...]` (비슷한 주제라 숫자도 비슷함)
- "프로젝트 일정 관리" → `[0.11, 0.67, -0.33, ...]` (다른 주제라 숫자가 다름)

의미가 비슷한 텍스트는 숫자 배열도 비슷해진다.
이 성질을 이용해서 **"질문과 의미가 가장 비슷한 조각"을 찾는 것**이 검색의 핵심이다.

두 숫자 배열이 얼마나 비슷한지 측정하는 방법을 **코사인 거리**라고 한다.
- 거리 = 0 → 완전히 같은 의미
- 거리 = 1 → 완전히 다른 의미
- 거리가 작을수록 더 관련 있는 조각이다

### 1.4 "Top-K"란?

검색 결과 중 **가장 비슷한 것 K개만 가져온다**는 뜻이다.

my-atlas에서는 PDF 청크 중 Top-2 (가장 비슷한 2개)를 가져온다.
72페이지 교재에서 54개 조각이 만들어졌는데, 질문할 때마다 54개를 다 보내는 게 아니라
**가장 관련 있는 2개만 골라서** AI에게 보낸다.

### 1.5 왜 청킹 품질이 중요한가?

AI는 우리가 전달한 조각만 읽을 수 있다.

**청킹이 잘 된 경우:**
```
질문: "테스팅의 7원칙이 뭐야?"
    → 검색 결과: [7원칙 전체가 담긴 청크] ← 이 조각 하나로 완벽한 답변 가능
    → AI 답변: 7원칙을 정확하게 모두 설명
```

**청킹이 잘못 된 경우 (현재 상태):**
```
질문: "테스팅의 7원칙이 뭐야?"
    → 검색 결과: [원칙 1만 담긴 45자짜리 조각], [원칙 7 + 다음 챕터 내용이 섞인 조각]
    → AI 답변: 원칙 1과 7만 언급하고 나머지 5개는 AI가 자기 지식으로 추측 (부정확할 수 있음)
```

**즉, 청킹이 잘못되면 아무리 좋은 AI를 써도 답변 품질이 떨어진다.**

## 2. 현재 ISTQB CTFL 청킹 현황

### 2.1 기본 정보

- 소스: `ISTQB_CTFL`
- 총 청크 수: 54개
- 청크 크기 범위: 26자 ~ 5,616자

### 2.2 사전 분석에서 발견된 문제

#### 문제 1: 목차가 본문으로 오인식됨

PDF 앞부분에는 목차가 있다:
```
제 1장 테스팅의 기초(Fundamentals of Testing) .................. 13
제 2장 소프트웨어 개발수명주기(SDLC)와 테스팅 .................. 25
```
시스템은 "제 1장"이라는 패턴을 보고 **"여기서부터 1장이 시작된다"** 고 판단했다.
하지만 이건 **목차일 뿐이고, 실제 1장 본문은 13페이지부터** 시작한다.
결과: 목차 한 줄짜리 의미없는 청크가 생성됨 (예: 148자)

#### 문제 2: 번호 매김 리스트가 각각 쪼개짐

교재에 이런 내용이 있다:
```
테스팅의 7원칙:
1. 테스팅은 결함의 존재를 밝히는 활동이지...
2. 완벽한 테스팅은 불가능하다...
3. 조기 테스팅으로 시간과 비용을 절약...
...
```
시스템은 "1. "로 시작하는 줄을 보고 **새로운 섹션이 시작된다**고 판단했다.
결과: 7원칙이 7개의 개별 청크로 찢어짐 (각각 45~213자의 한 문장짜리 조각)

#### 문제 3: 같은 이름의 청크가 여러 개

"ISTQB_CTFL - chunk - 001" 이라는 이름의 청크가 **7개나** 존재한다.
챕터가 바뀔 때마다 번호가 1번부터 다시 시작되기 때문이다.
나중에 검색 결과를 보더라도 이 조각이 어느 챕터 내용인지 구분할 수 없다.

#### 문제 4: 조각 크기 편차가 극심

| 가장 작은 청크 | 가장 큰 청크 | 차이 |
|---------------|-------------|------|
| 26자 (6장 목차 한 줄) | 5,616자 (7장 References 전체) | **216배** |

설정상 500~800 단어 범위로 자르도록 되어 있지만, 실제로는 지켜지지 않고 있다.

## 3. 테스트 전략

청킹 품질을 검증하기 위해 **3단계**로 나누어 테스트한다.
앞 단계를 통과해야 다음 단계로 넘어간다.

```
[Stage 1] 조각 품질 검증
          → 저장된 조각 자체가 올바른 크기와 내용을 갖고 있는가?
          → DB 쿼리로 확인 (자동화 가능)
              ↓
[Stage 2] 검색 정확도 검증
          → 질문을 했을 때, 정답이 담긴 조각이 검색 결과에 포함되는가?
          → Senior 채팅에서 직접 질문하여 확인
              ↓
[Stage 3] 답변 품질 검증
          → 최종 답변이 교재 원문과 대조했을 때 정확하고 빠짐없는가?
          → ISTQB 원문과 대조하여 채점
```

## 4. Stage 1 — 조각 품질 검증

> 목적: 저장된 54개 조각이 "의미 있는 단위"로 나뉘어져 있는지 확인한다.

### 4.1 좋은 조각의 기준

| 기준 | 설명 | 합격 조건 |
|------|------|-----------|
| 최소 크기 | 한 문장짜리 조각은 의미 전달이 어렵다 | 모든 조각 ≥ 200자 |
| 최대 크기 | 너무 큰 조각은 여러 주제가 섞일 수 있다 | 모든 조각 ≤ 5,000자 |
| 의미 완결성 | 조각 하나만 읽어도 무슨 내용인지 이해 가능해야 한다 | 수동 검토 |
| 이름 고유성 | 같은 이름의 조각이 있으면 구분이 안 된다 | 동일 이름 조각 0개 |
| 노이즈 없음 | 목차, 저작권, 페이지 번호 등은 답변에 쓸모없다 | 노이즈 전용 조각 0개 |

### 4.2 검증 방법: DB 쿼리

Docker 컨테이너가 실행 중인 상태에서, 아래 쿼리를 실행한다.

```bash
# 터미널에서 실행하는 방법
docker exec myqaweb-db psql -U myqaweb -d myqaweb -c "여기에 SQL 쿼리"
```

#### Q1. 크기 분포 확인

```sql
SELECT 
    COUNT(*) as 전체_조각수,
    MIN(LENGTH(content)) as 최소_글자수,
    MAX(LENGTH(content)) as 최대_글자수,
    AVG(LENGTH(content))::int as 평균_글자수,
    COUNT(*) FILTER (WHERE LENGTH(content) < 200) as 너무_작은_조각,
    COUNT(*) FILTER (WHERE LENGTH(content) > 5000) as 너무_큰_조각
FROM knowledge_base 
WHERE source = 'ISTQB_CTFL';
```

- `너무_작은_조각`이 0이 아니면 → 청킹에 문제가 있다
- `너무_큰_조각`이 0이 아니면 → 청킹이 제대로 분할하지 못한 것이다

#### Q2. 너무 작은 조각 목록 확인

```sql
SELECT id, title, LENGTH(content) as 글자수, content
FROM knowledge_base 
WHERE source = 'ISTQB_CTFL' AND LENGTH(content) < 200
ORDER BY LENGTH(content);
```

이 결과를 보고 **왜 이렇게 작은 조각이 생겼는지** 원인을 분석한다.
(예: 목차 한 줄, 번호 리스트의 한 항목 등)

#### Q3. 같은 이름 조각 확인

```sql
SELECT title, COUNT(*) as 중복_횟수
FROM knowledge_base 
WHERE source = 'ISTQB_CTFL'
GROUP BY title 
HAVING COUNT(*) > 1
ORDER BY COUNT(*) DESC;
```

중복이 있으면 → 검색 결과에서 어떤 조각인지 구분할 수 없으므로 개선 필요

#### Q4. 임베딩 누락 확인

```sql
SELECT id, title 
FROM knowledge_base 
WHERE source = 'ISTQB_CTFL' AND embedding IS NULL;
```

임베딩이 없는 조각은 **검색 자체가 불가능**하다.
숫자 배열이 없으면 "의미가 비슷한 것"을 비교할 수 없기 때문이다.

### 4.3 판정 기준

| 결과 | 판정 | 다음 행동 |
|------|------|-----------|
| 너무 작은 조각 = 0, 중복 이름 = 0, 임베딩 누락 = 0 | **PASS** | Stage 2로 진행 |
| 위 중 하나라도 0이 아님 | **FAIL** | 청킹 로직 개선 후 재업로드 → Stage 1 재실행 |

## 5. Stage 2 — 검색 정확도 검증

> 목적: 질문을 했을 때, **정답이 들어있는 조각이 실제로 검색되는지** 확인한다.

### 5.1 왜 이 테스트가 필요한가?

Stage 1에서 조각의 크기와 형식이 괜찮다고 확인되었더라도,
**질문과 관련 없는 조각이 검색될 수 있다.**

예를 들어:
- "경계값 분석"을 질문했는데, "테스트 관리" 조각이 검색될 수 있다
- 이유: 두 텍스트 모두 "테스트"라는 단어를 포함하고 있어서 임베딩이 비슷할 수 있음

이런 문제를 찾기 위해, **정답을 알고 있는 질문**을 던져서 올바른 조각이 나오는지 확인한다.

### 5.2 테스트 설계 원칙

ISTQB CTFL은 챕터 구조가 명확하다:
- 1장: 테스팅의 기초
- 2장: SDLC와 테스팅
- 3장: 정적 테스팅
- 4장: 테스트 분석과 설계
- 5장: 테스트 활동 관리
- 6장: 테스트 도구

따라서 **"이 질문의 정답은 N장에 있다"** 는 것을 미리 알 수 있다.
검색된 조각이 해당 챕터의 내용을 포함하고 있으면 검색이 정확한 것이다.

### 5.3 테스트 케이스

| # | 질문 | 정답이 있는 챕터 | 확인 포인트 |
|---|------|-----------------|-------------|
| R-01 | "테스팅의 7가지 원리를 설명해줘" | 1장 (1.3절) | 7원칙이 빠짐없이 답변에 포함되는가? (현재 7개로 쪼개져 있어 위험) |
| R-02 | "V-모델에서 각 테스트 레벨의 역할은?" | 2장 (2.2절) | 컴포넌트/통합/시스템/인수 테스트가 모두 언급되는가? |
| R-03 | "정적 테스팅과 동적 테스팅의 차이는?" | 3장 (3.1절) | 정적=코드 실행 없이 / 동적=코드 실행하며 — 이 구분이 명확한가? |
| R-04 | "동등 분할과 경계값 분석의 차이를 설명해줘" | 4장 (4.2절) | 두 기법의 정의와 차이가 교재 내용과 일치하는가? |
| R-05 | "테스트 추정 기법에는 어떤 것이 있어?" | 5장 (5.1절) | 경험 기반 / 메트릭 기반 추정이 언급되는가? |
| R-06 | "테스트 자동화의 장단점은?" | 6장 | 자동화 장점과 리스크가 균형있게 설명되는가? |
| R-07 | "리스크 기반 테스팅이란 무엇이야?" | 5장 (5.2절) | 제품 리스크와 프로젝트 리스크가 구분되어 설명되는가? |
| R-08 | "리뷰 프로세스의 단계를 설명해줘" | 3장 (3.2절) | 계획→착수→개별 리뷰→이슈 논의→수정 순서가 나오는가? |
| R-09 | "테스트 케이스의 구성 요소는?" | 4장 or 5장 | 전제조건, 입력, 기대결과 등이 언급되는가? |
| R-10 | "회귀 테스팅은 왜 필요한가?" | 2장 (2.3절) | 변경으로 인한 의도치 않은 부작용 검출이 핵심 이유로 나오는가? |

### 5.4 테스트 실행 방법

1. 브라우저에서 `http://localhost:5173/senior` 접속
2. 위 표의 질문을 하나씩 입력
3. AI 답변을 읽고, **정답이 있는 챕터의 내용이 답변에 반영되었는지** 확인
4. 아래 기준으로 판정

| 판정 | 기준 | 예시 |
|------|------|------|
| **PASS** | 해당 챕터의 핵심 내용이 답변에 정확히 포함됨 | R-01에서 7원칙이 모두 정확하게 나옴 |
| **PARTIAL** | 일부만 맞거나, 내용이 불완전함 | R-01에서 원칙 3개만 나오고 나머지는 누락 |
| **FAIL** | 엉뚱한 내용이거나, "모르겠다"고 답변 | R-01에서 테스팅 원리 대신 테스트 관리 내용이 나옴 |

### 5.5 결과 기록 양식

| # | 질문 | 기대 챕터 | 판정 | 비고 (어떤 내용이 맞았거나 빠졌는지) |
|---|------|-----------|------|--------------------------------------|
| R-01 | 테스팅의 7가지 원리 | 1장 | | |

## 6. Stage 3 — 답변 품질 검증

> 목적: AI 답변의 **정확성과 완성도**를 교재 원문과 대조하여 평가한다.

### 6.1 왜 Stage 2와 별도로 필요한가?

Stage 2에서는 "맞는 조각이 검색되었는가?"를 확인했다.
하지만 맞는 조각이 검색되었더라도:
- AI가 조각의 내용을 **잘못 해석**할 수 있다
- AI가 조각에 없는 내용을 **지어낼** 수 있다 (이것을 "할루시네이션"이라 부른다)
- 조각에 담긴 내용이 **불완전해서** 답변도 불완전할 수 있다

### 6.2 평가 기준

| 기준 | 설명 | 점수 |
|------|------|------|
| **정확성** | AI 답변 내용이 ISTQB 교재 원문과 일치하는가? | 0~3점 |
| **완성도** | 질문에 대한 답변이 빠짐없이 포함되어 있는가? | 0~3점 |
| **지어낸 내용 없음** | ISTQB 교재에 없는 내용을 AI가 만들어내지 않았는가? | 0~3점 |
| **읽기 쉬움** | 답변이 체계적으로 정리되어 있는가? | 0~1점 |

**총점: 10점 만점**

점수 부여 예시:
- 정확성 3점: 교재 원문과 완전히 일치
- 정확성 2점: 대부분 맞지만 일부 부정확
- 정확성 1점: 맞는 내용과 틀린 내용이 섞여 있음
- 정확성 0점: 대부분 틀림

### 6.3 테스트 실행 방법

1. Stage 2에서 사용한 동일한 질문 10개를 사용한다
2. AI 답변을 받은 후, **ISTQB CTFL 교재 원문**(해당 챕터)을 열어서 대조한다
3. 위 4가지 기준으로 채점한다

| # | 질문 | 정확성(/3) | 완성도(/3) | 지어낸 내용 없음(/3) | 읽기 쉬움(/1) | 총점(/10) | 비고 |

| R-06 | 테스트 자동화 장단점 | | | | | | |

## 7. 문제 발생 시 개선 방안

각 Stage에서 실패했을 때 어떤 부분을 고쳐야 하는지 정리한다.

### 7.1 Stage 1 실패 — 조각 자체에 문제가 있는 경우

| 증상 | 원인 | 개선 방법 |
|------|------|-----------|
| 45자, 80자 등 초소형 조각이 많다 | 번호 리스트(`1. xxx`)를 챕터 시작으로 오인식 | 섹션 구분 패턴(정규식)을 수정하여, "제 N장", "Chapter N"만 챕터로 인식하도록 변경 |
| 목차 내용이 별도 조각으로 존재 | 목차 페이지의 "제 1장..." 텍스트가 챕터로 인식됨 | PDF 텍스트에서 목차 영역을 제거하는 전처리 추가 |
| "chunk - 001" 이름이 7개 중복 | 챕터가 바뀔 때마다 번호가 리셋됨 | 전체 PDF에서 번호를 연속으로 매기거나, 챕터명을 이름에 포함 |
| 5,000자 이상 초대형 조각 존재 | 청킹 로직이 해당 구간에서 분할에 실패 | 최대 크기 초과 시 강제 분할하는 로직 추가 |

### 7.2 Stage 2 실패 — 엉뚱한 조각이 검색되는 경우

| 증상 | 원인 | 개선 방법 |
|------|------|-----------|
| 4장 내용을 물어봤는데 5장 조각이 나옴 | 두 챕터의 내용이 비슷한 단어를 공유 | 각 조각에 "이 내용은 N장 OO절의 내용이다"라는 문맥 정보를 추가 |
| 저작권/목차 조각이 검색됨 | 노이즈 조각에도 임베딩이 생성되어 검색 대상에 포함됨 | 노이즈 조각을 DB에서 제거하거나, 임베딩을 NULL로 설정하여 검색 제외 |
| 한 질문에 같은 내용의 조각 2개가 나옴 | 목차 + 본문에서 같은 텍스트가 중복 청킹됨 | 중복 조각 정리 |

### 7.3 Stage 3 실패 — 검색은 맞지만 답변이 부정확한 경우

| 증상 | 원인 | 개선 방법 |
|------|------|-----------|
| 답변에 교재 내용의 일부만 포함됨 | 하나의 개념이 여러 조각에 쪼개져 있어서, 검색된 2개에 전체가 담기지 않음 | 조각 크기를 늘리거나, 검색 개수(Top-K)를 2→3으로 증가 |
| AI가 교재에 없는 내용을 지어냄 | 조각 내용이 불충분해서 AI가 자체 지식으로 보충 | 조각에 더 많은 문맥 포함, 또는 AI 프롬프트에 "주어진 자료에만 기반하여 답변하라" 강화 |
| 답변이 교재 원문과 다름 | 조각 내에 여러 주제가 섞여 있어 AI가 잘못 해석 | 조각을 하나의 주제 단위로 더 세밀하게 분리 |

## 8. 전체 테스트 흐름 요약

```
Step 1. Stage 1 — DB 쿼리 4개 실행 → 조각 품질 확인
        ├→ FAIL: 청킹 로직 개선 → 재업로드 → Step 1 다시
        └→ PASS: 다음으로
            ↓
Step 2. Stage 2 — Senior 채팅에서 질문 10개 → 검색 정확도 확인
        ├→ FAIL (PASS < 7개): 실패한 질문 분석 → 해당 조각 확인 → 개선
        └→ PASS (PASS ≥ 7개): 다음으로
            ↓
Step 3. Stage 3 — 답변을 교재 원문과 대조 → 채점
        ├→ FAIL (평균 < 7.0): 낮은 점수 항목 분석 → 개선
        └→ PASS (평균 ≥ 7.0): 청킹 품질 최종 승인
```
