# Docker Compose DB 공유 아키텍처 개선

## 1. 문제

### 현상
- knowledge-base worktree에서 `GET /api/kb` 호출 시 500 에러
- PostgreSQL TOAST 데이터 손상: `missing chunk number 1 for toast value 17037`
- 데이터 파일 누락: `could not open file "base/16384/16995": No such file or directory`
- DDL 명령(CREATE TABLE) 무한 hang

### 근본 원인: 4개의 PostgreSQL 인스턴스가 동일 볼륨을 공유

```
docker volume: my-atlas_pgdata (1개)
     ↑
     ├── myqaweb-db                      (포트 5432) ← 동시 쓰기
     ├── myqaweb-knowledge-base-db       (포트 5434) ← 동시 쓰기
     ├── myqaweb-feature-registry-db     (포트 5433) ← 동시 쓰기
     └── myqaweb-my-senior-db            (포트 5435) ← 동시 쓰기
```

각 worktree의 `docker-compose.yml`이 독립적으로 DB 컨테이너를 띄우지만, 볼륨 이름이 `my-atlas_pgdata`로 하드코딩되어 있어 **4개의 PostgreSQL 프로세스가 같은 데이터 파일을 동시에 읽고 쓴다.**

PostgreSQL은 **단일 프로세스가 데이터 디렉토리를 독점**하도록 설계되어 있다. 여러 인스턴스가 같은 파일을 쓰면:
- TOAST 청크 덮어쓰기 → **데이터 손상**
- WAL 충돌 → **트랜잭션 꼬임**
- Lock 파일 경합 → **DDL hang**

## 2. 해결 방안

### 원칙: DB 컨테이너 1개, Backend 컨테이너 N개

DB를 공유하되, **PostgreSQL 인스턴스는 1개만** 실행한다. 모든 worktree의 backend가 그 하나의 DB에 접속한다.

### 변경 후 아키텍처

```
[myqaweb-db]  ← 포트 5432, 메인 레포에서만 실행
     ↑ (host.docker.internal:5432)
     ├── myqaweb-backend                   (포트 8080, 메인)
     ├── myqaweb-knowledge-base-backend    (포트 8082)
     ├── myqaweb-feature-registry-backend  (포트 8081)
     └── myqaweb-my-senior-backend         (포트 8083)
```

## 3. 구현 계획

### Step 1. 메인 레포 docker-compose.yml — DB 전용 프로파일 분리

DB 서비스를 항상 실행되는 기본 서비스로 유지하되, 다른 worktree에서도 접근 가능하도록 호스트 포트 5432를 고정한다.

```yaml
# 메인 레포 docker-compose.yml — 변경 없음 (DB + Backend + Frontend 모두 포함)
services:
  db:
    image: pgvector/pgvector:pg15
    container_name: myqaweb-db
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    # ... (기존 설정 유지)

volumes:
  pgdata:
    name: my-atlas_pgdata
```

### Step 2. worktree용 docker-compose.override.yml 작성

각 worktree에서는 **DB 서비스를 제거**하고, backend가 호스트의 DB에 접속하도록 변경한다.

```yaml
# .claude/worktrees/*/docker-compose.yml (worktree용)
version: "3.9"

services:
  # DB 서비스 없음 — 메인 레포의 myqaweb-db 사용

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: ${COMPOSE_CONTAINER_PREFIX}-backend
    restart: unless-stopped
    environment:
      # host.docker.internal로 호스트의 DB 컨테이너에 접속
      SPRING_DATASOURCE_URL: jdbc:postgresql://host.docker.internal:5432/myqaweb
      SPRING_DATASOURCE_USERNAME: ${SPRING_DATASOURCE_USERNAME:-myqaweb}
      SPRING_DATASOURCE_PASSWORD: ${SPRING_DATASOURCE_PASSWORD:-admin}
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      OPENAI_API_KEY: ${OPENAI_API_KEY}
    ports:
      - "${BACKEND_PORT}:8080"
    extra_hosts:
      - "host.docker.internal:host-gateway"
    volumes:
      - ./logs:/app/logs

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    container_name: ${COMPOSE_CONTAINER_PREFIX}-frontend
    restart: unless-stopped
    environment:
      VITE_API_BASE_URL: http://localhost:${BACKEND_PORT}
    ports:
      - "${FRONTEND_PORT}:5173"
    volumes:
      - ./frontend/src:/app/src
    depends_on:
      - backend
```

### Step 3. 각 worktree의 .env 수정

DB 관련 설정을 호스트 DB 접속으로 변경한다.

```bash
# .env (worktree 공통 변경)
# DB_PORT 제거 (worktree에서 DB 컨테이너를 띄우지 않으므로)
SPRING_DATASOURCE_URL=jdbc:postgresql://host.docker.internal:5432/myqaweb

# 기존 유지
COMPOSE_CONTAINER_PREFIX=myqaweb-knowledge-base
BACKEND_PORT=8082
FRONTEND_PORT=5175
```

### Step 4. 운영 절차 문서화

```markdown
## Worktree Docker 운영 규칙

1. `docker compose up -d`로 DB를 먼저 띄우는 곳은 **메인 레포만**
2. worktree에서는 메인 DB가 떠 있는 상태에서 `docker compose up -d` 실행
3. 메인 DB가 안 떠 있으면 worktree backend가 접속 실패함
4. 시작 순서: 메인 DB → worktree backend/frontend
5. 종료 순서: worktree → 메인 DB (마지막에)
```

### Step 5. 기존 손상 복구

1. 모든 worktree의 DB 컨테이너 중지
2. 메인 레포의 DB 컨테이너만 남기고 나머지 제거
3. 손상된 볼륨 복구 (VACUUM FULL 또는 볼륨 재생성 + Flyway 재실행)

**`!reset` YAML 태그**: Docker Compose의 ports merge 문제를 해결하기 위해 `ports: !reset`으로 base의 포트 매핑을 완전히 대체.
